Introduction ---介绍
2-0270280080100808905FboA0oa8e0o ---2-0270280080100808905FboA0oa8e0
The purpose of the ISO/IEC/IEEE 29119 series of software testing standards is to define an internationally- agreed set of standards for software testing that can be used by any organization when managing or performing any form of software testing. ---ISO/IEC/IEEE 29119系列软件测试标准的目的是定义一套国际商定的软件测试标准，任何组织在管理或执行任何形式的软件测试时都可以使用这些标准。
This part of ISO/IECIEEE 29119 defines a unified approach for describing test cases in a modular way, which assists with the creation of items like keyword-driven test specifications and test automation frameworks. The term "keyword" refers to the elements which are, once defined, used to compose test cases, such as with building blocks. ISO/IEC/IEEE 29119-5 will explain the main concepts and application of Keyword-Driven Testing. It will also define attributes of frameworks designed to support Keyword-Driven Testing. ---ISO/IECIEEE 29119的这一部分定义了以模块化方式描述测试用例的统一方法，这有助于创建关键字驱动的测试规范和测试自动化框架等项目。术语“关键字”指的是定义后用于组成测试用例的元素，例如构建块。ISO/IEC/IEEE 29119-5将解释关键词驱动测试的主要概念和应用。它还将定义为支持关键字驱动测试而设计的框架的属性。
The concepts and definitions relating to software testing defined in ISO/IEC/IEEE 29119-1 are also applicable to ISO/IEC/IEEE 29119-5. ---ISO/IEC/IEEE 29119-1中定义的与软件测试相关的概念和定义也适用于ISO/IEC/IEEE 29119-5。
The test process model on which the Keyword-Driven Testing framework is based is defined in ISO/IEC/IEEE 29119-2 Test Processes. It comprises test process descriptions that define the software testing processes at the organizational level, test management level and dynamic test level. Supporting informative diagrams describing the processes are also provided in ISO/IEC/IEEE 29119-2. 5 Introduction to Keyword-Driven Testing ---5关键词驱动测试简介
5.1 Overview ---5.1概述
Keyword-Driven Testing is a test case specification approach that is commonly used to support test automation and the development of test automation frameworks. However, it can also be used if no automation approach is planned or established. ---关键词驱动测试是一种测试用例规范方法，通常用于支持测试自动化和测试自动化框架的开发。然而，如果没有计划或建立自动化方法，也可以使用它。
In principle, Keyword-Driven Testing can be applied at all testing levels (e.g. component testing, system testing) and for various types of testing (e.g. functional testing, reliability testing). Keyword-Driven Testing benefits include the following: ---原则上，关键字驱动测试可以应用于所有测试级别（例如组件测试、系统测试）和各种类型的测试（例如功能测试、可靠性测试）。关键字驱动测试的好处包括：
-ease of use ----易于使用
- understandability ----可理解性
-maintainability ----可维护性
test information reuse ---测试信息重用
● ---● 
support of test automation ---测试自动化支持
6.08二 ---6.08二 
-potential cost and schedule savings ----潜在的成本和进度节约
The fundamental idea in Keyword-Driven Testing is to provide a set of "building blocks", referred to as keywords, that can be used to create manual or automated test cases without requiring detailed knowledge of programming or test tool expertise. The ultimate goal is to provide a basic, unambiguous set of keywords comprehensive enough so that most, if not all, required test cases can be entirely composed of these keywords. The vocabulary included in these dictionaries or libraries of keywords is, therefore, a reflection of the language and level of abstraction used to write the test cases, and not of any standard computer programming language. ---关键字驱动测试的基本思想是提供一组“构建块”（称为关键字），可以用于创建手动或自动测试用例，而不需要详细的编程知识或测试工具专业知识。最终目标是提供一组基本的、明确的、足够全面的关键字，以便大多数（如果不是所有）所需的测试用例可以完全由这些关键字组成。因此，这些字典或关键字库中包含的词汇反映了用于编写测试用例的语言和抽象级别，而不是任何标准的计算机编程语言。
For test automation, each keyword needs to be implemented in software. ---对于测试自动化，每个关键字都需要在软件中实现。

-At a low level, each keyword is associated with a detailed set of one or more actions that describe the exact steps that are to be performed. ----在低级别，每个关键字都与一组详细的一个或多个动作相关联，这些动作描述了要执行的确切步骤。

— At a high level, a meaningful name is used to identify the keyword. This keyword may require a set of input parameters, which would also belong to this level in the structure. The keyword and the parameters together comprise a high-level description of the actions associated with a test case. ----在较高级别上，使用有意义的名称来标识关键字。此关键字可能需要一组输入参数，这些参数也属于结构中的此级别。关键字和参数一起包括与测试用例相关联的动作的高级描述。
Thus, instead of describing each individual action in test cases, tests can be defined at a higher level of abstraction using keywords. Higher levels of abstraction can be achieved by using composite keywords, which are comprised of other keywords to describe associated actions. ---因此，可以使用关键字在更高的抽象级别定义测试，而不是在测试用例中描述每个单独的动作。通过使用复合关键字可以实现更高级别的抽象，复合关键字由其他关键字组成，用于描述相关的动作。
An example of the benefits obtained from both manual and automated keyword-driven test cases is enhanced maintainability. Consider a case where the precise set of actions to carry out a commonly repeated operation has changed. The modularity introduced by keywords allows modification of only the actions for the changed operation in the relevant lower-level keyword, leaving the test cases untouched. Without modularity, it may be necessary to modify each occurrence of this operation in all of the test cases. ---从手动和自动关键字驱动测试用例中获得的好处的一个例子是增强的可维护性。考虑一种情况，即执行通常重复操作的精确操作集发生了变化。关键字引入的模块化只允许修改相关较低级别关键字中更改操作的操作，而不影响测试用例。如果没有模块化，可能需要在所有测试用例中修改此操作的每次出现。
Modularization has helped popularize this approach. If test automation is required, a framework can be created to interpret manually created keyword test cases as executable test automation scripts. This is achieved by implementing test automation code for each keyword (e.g. keyword execution code). ---模块化有助于普及这种方法。如果需要测试自动化，可以创建一个框架来将手动创建的关键字测试用例解释为可执行的测试自动化脚本。这通过为每个关键字实现测试自动化代码（例如关键字执行代码）来实现。

5.2 Layers in Keyword-Driven Testing ---5.2关键词驱动测试中的层
5.2.1 Overview ---5.2.1概述
Keywords can represent actions at different abstraction levels. For example, one keyword can refer to a very complex set of activities, like the creation of a contract, which includes a lot of steps, while another keyword can refer to a very simple action, like pressing a button on a graphical user interface. The first keyword is close to the business and end user domain, while the second is closer to the test interface. Keywords that are written at a similar level of detail, and have a similar relationship to the stakeholder's view, are said to belong to the same abstraction layer. ---关键字可以表示不同抽象级别的操作。例如，一个关键字可以指一组非常复杂的活动，如创建合同，其中包含许多步骤，而另一个关键字则可以指一个非常简单的操作，如按下图形用户界面上的按钮。第一个关键字接近业务和最终用户领域，而第二个关键字更接近测试界面。以类似的细节级别编写的关键字，以及与涉众的视图具有类似关系的关键字，被称为属于同一抽象层。
Keyword-Driven Testing can be organized by using one or more layers. Typical layers are the end user domain layer and the test interface layer. ---关键词驱动测试可以通过使用一个或多个层来组织。典型的层是最终用户域层和测试接口层。
While many implementations of Keyword-Driven Testing will comprise two or three abstraction layers, in some cases it may be necessary to structure keywords in more layers. ---虽然关键字驱动测试的许多实现将包括两个或三个抽象层，但在某些情况下，可能需要在更多层中构造关键字。
The topmost layer is the most abstract layer, which is generally aligned with the wording of the application's users. In practice, the topmost layer is usually the domain layer. However, in some situations the domain layer may not be required, and another, more abstract layer is used (e.g. if the test cases are supposed to span several different end user domains, a meta domain layer can be introduced). ---最顶层是最抽象的层，通常与应用程序用户的措辞一致。实际上，最顶层通常是域层。然而，在某些情况下，可能不需要域层，而是使用另一个更抽象的层（例如，如果测试用例要跨越多个不同的最终用户域，则可以引入元域层）。
The lowest layer is the most detailed layer. It is commonly aligned with the names of test interface elements (e.g. "selectMenultem"). In practice, this layer is usually, but not always, the test interface layer (e.g. as sometimes a test interface layer is not required, or for specific reasons, even more detailed layers may be used). ---最底层是最详细的层。它通常与测试接口元素的名称对齐（例如“selectMenultem”）。在实践中，该层通常是但不总是测试接口层（例如，有时不需要测试接口层，或者出于特定原因，可以使用更详细的层）。
Most Keyword-Driven Test systems will have more than one layer due to factors such as having understandable keyword test cases, maintainability and division of work relying on a multi-layer structure. If only one layer is implemented, it will commonly be either at a low level, which affects the readability of the keyword test cases, or at a high level, which can result in more keyword execution code. ---由于诸如具有可理解的关键字测试用例、可维护性和依赖于多层结构的工作分工等因素，大多数关键字驱动测试系统将具有不止一层。如果只实现了一个层，它通常要么处于低级别，这会影响关键字测试用例的可读性，要么处于高级别，这可能会导致更多的关键字执行代码。

The test framework can be designed to handle blocked keywords on the test item. Keywords can then be optionally marked either as “may be blocked" or as "must not be blocked". In the first case, a blocking (unsuccessful execution) of the keyword would not affect the test result; in the second case, the test result will be affected. A keyword can be marked either globally (the property is default for all applications in test cases) or overridden when it is used in a test case. ---测试框架可以被设计为处理测试项上被阻止的关键字。然后，可以选择将关键字标记为“可能被阻止”或“不能被阻止”。在第一种情况下，关键字的阻止（不成功执行）不会影响测试结果；在第二种情况下会影响测试结果。可以全局标记关键字（测试用例中所有应用程序的默认属性）或在测试用例中使用时重写。
The test framework can additionally provide an eror recognition mechanism that can take care of errors returned by a keyword. Failures can be logged and described as clearly as possible in order to simplify the correction of errors in the automation framework and investigate its cause, which may be a software-defect. ---测试框架还可以提供一个eror识别机制，该机制可以处理关键字返回的错误。可以尽可能清楚地记录和描述故障，以简化自动化框架中错误的纠正并调查其原因，这可能是软件缺陷。
5.4 Keywords and Data ---5.4关键词和数据
Keyword-Driven Testing can be enhanced if keywords are associated with data. To allow an association with data, in many cases keywords will need to have parameters which may be fixed, or list driven. ---如果关键字与数据关联，则可以增强关键字驱动测试。为了允许与数据关联，在许多情况下，关键字需要具有固定的或列表驱动的参数。
Most keywords will need to have at least one parameter to specify the object they apply to. Some will need another parameter to specify input, (e.g., true/false, a string to type, an option to select in a combo box). This input will generally depend on the type of control and the type of action. ---大多数关键字需要至少一个参数来指定它们应用的对象。有些关键字需要另一个参数指定输入（例如，true/false、要键入的字符串、要在组合框中选择的选项）。该输入通常取决于控制类型和动作类型。

6.4 Keywords and data-driven testing ---6.4关键词和数据驱动测试
Keywords combined with parameters and separate data sets for these parameters (e.g. data-driven) may offer improved testing. Data-driven testing can be applied when the same sequence of keyword actions are to be used with different sets of data. In this combined approach, the data can be stored separately from the keyword test cases. The data can be stored in constructs such as tables, databases, or real-time generators, and is then read into the keyword test cases. The repeated keyword sequence using different data, in effect, creates new tests. ---结合参数的关键字和这些参数的单独数据集（例如数据驱动）可以提供改进的测试。当相同的关键字操作序列用于不同的数据集时，可以应用数据驱动测试。在这种组合方法中，数据可以与关键字测试用例分开存储。数据可以存储在表、数据库或实时生成器等结构中，然后读入关键字测试用例。使用不同数据的重复关键字序列实际上会创建新的测试。
EXAMPLE Data-driven testing is useful for multi-lingual testing or internationalization testing, where the same test cases are to be performed at user interfaces but with different languages. It should be kept in mind that not all internationalization issues are easily addressed by data-driven testing: in the case of lexicographical sorting, the requested item may have not only have a different label but also a different position. ---示例：数据驱动测试对于多语言测试或国际化测试非常有用，其中相同的测试用例将在用户界面上执行，但使用不同的语言。应该记住，并不是所有的国际化问题都可以通过数据驱动的测试轻松解决：在词典排序的情况下，所请求的项目可能不仅具有不同的标签，而且具有不同的位置。

6.5 Modularity and refactoring ---6.5模块化和重构
Modularity in Keyword-Driven Testing is used to improve the longevity of the test cases. However, with the passage of time, changes in the test item, new test cases or new people on the team can all lead to maintenance issues. ---关键词驱动测试中的模块化用于提高测试用例的寿命。然而，随着时间的推移，测试项目、新的测试用例或团队中的新人的变化都可能导致维护问题。
Possible issues are as follows: ---可能的问题如下：

- Redundant keywords: where two or more keywords for the same objective come into existence. ----冗余关键词：同一目标的两个或多个关键词出现。

Conflicts where changes in keywords(e.g. structure or semantic), which fix an issue in a number of test cases, create new issues in other test cases. This has associated cost factors. ---关键字（例如，结构或语义）的变化在许多测试用例中解决了问题，在其他测试用例中产生了新问题。这有相关的成本因素。

To avoid these issues, the following maintenance actions should be considered: ---为避免这些问题，应考虑以下维护措施：
A framework for Keyword-Driven Testing should provide a way of creating a cross-reference for the used keywords, allowing identification of which keywords are used in which places and how frequently they are used. This shows if, and how much, a change in a keyword will affect existing test cases. ---关键词驱动测试框架应提供一种为所用关键词创建交叉引用的方法，允许识别在哪些地方使用哪些关键词以及使用频率。这显示了关键字的更改是否会影响现有测试用例，以及影响的程度。
In some organizations, an authority is required who is responsible for all keywords, additions and any changes to existing keywords or how they are used. That authority assures consistency throughout the project including both development and testing stages. ---在一些组织中，需要一个权威机构负责所有关键字、添加和对现有关键字的任何更改或如何使用它们。该权威确保了整个项目的一致性，包括开发和测试阶段。
On a regular basis (e.g. once a month), a keyword review meeting can be held. At this meeting, testers can decide about the introduction or modification of keywords and discuss the structure of the keywords. If an authority is in charge of keywords, they should also be in attendance. ---可以定期（例如每月一次）召开关键词审查会议。在这个会议上，测试人员可以决定关键词的引入或修改，并讨论关键词的结构。如果权威机构负责关键字，他们也应该出席。
A clear structure to document the keywords should be produced. Keywords can be grouped by layer, test item and region in the test item (e.g. dialog, objective or others). Keywords that are supposed to be usable by all testers will normally be stored separately from keywords which are only useful for a limited number of people. ---应该制作一个清晰的结构来记录关键词。关键字可以按层、测试项和测试项中的区域（例如对话框、目标或其他）分组。假设所有测试人员都可以使用的关键字通常与只对有限数量的人有用的关键字分开存储。
Keywords can be subjected to configuration management practices(e.g. the authority mentioned above). The ability to change keywords would normally be limited to those who need to do changes or the authority. All changes need to be well documented. Access to prior versions (e.g. the option for rolling back)should also be provided. ---关键字可以服从配置管理实践（例如上述权限）。更改关键字的能力通常仅限于那些需要更改的人或权限。所有变更都需要做好记录。还应提供对先前版本的访问（例如回滚选项）。

6.7 Converting non keyword-driven test cases into Keyword-Driven Testing ---6.7将非关键字驱动测试用例转换为关键字驱动测试
If Keyword-Driven Testing is to be introduced into an existing project, existing test cases can be converted into keyword test cases. ---如果要在现有项目中引入关键字驱动测试，则可以将现有测试用例转换为关键字测试用例。
In addition to the advantages of Keyword-Driven Testing, reasons for deciding to convert existing test cases into Keyword-Driven test cases include the following: ---除了关键字驱动测试的优点之外，决定将现有测试用例转换为关键字驱动测试用例的原因如下：
a)Uniformity: ensuring that all test cases have a similar structure and style will enhance readability, maintainability and so reduce costs. Future maintenance may be cheaper if only one style of test case is to be maintained. ---a） 一致性：确保所有测试用例具有相似的结构和样式，将增强可读性、可维护性，从而降低成本。如果只维护一种类型的测试用例，未来的维护可能会更便宜。
b)Efficiency: keywords identified from existing test cases may be reusable in future test cases. ---b） 效率：从现有测试用例中识别的关键字可以在未来的测试用例中重用。
c)Automation: the same automation framework may be used for old and new test cases. ---c） 自动化：相同的自动化框架可以用于新旧测试用例。
d)Understandability: test cases may be used and maintained by non-technical testers (often with business knowledge). ---d） 可理解性：非技术测试人员（通常具有业务知识）可以使用和维护测试用例。
Reasons to keep existing test cases and not convert them into Keyword-Driven test cases include the following: ---保留现有测试用例而不将其转换为关键字驱动测试用例的原因包括：
-The number of existing test cases is large compared with the number of additional test cases that are needed. ----与所需的额外测试用例的数量相比，现有测试用例的数目很大。
- There is proven and maintainable automation for the existing test cases. ----对于现有的测试用例，有经过验证和可维护的自动化。

The cost of converting the test cases is expected to exceed the benefits. ---转换测试用例的成本预计将超过收益。
The decision to move to Keyword-Driven Testing for existing projects should be carefully considered. ---对现有项目进行关键词驱动测试的决定应仔细考虑。

B.1 General benefits of Keyword-Driven Testing ---B、 1关键字驱动测试的一般好处
By composing all of the test cases from a fixed and defined set of keywords, the benefits can include the following: ---通过从一组固定和定义的关键字组成所有测试用例，其好处包括：
- Keywords can be defined in natural language meaning that, test cases can be written with more or less detail, depending on the project's needs. ----关键词可以用自然语言定义，也就是说，根据项目的需要，测试用例可以写得更详细或更详细。
- Test cases become clear and understandable. This supports efficient manual test execution. ----测试用例变得清晰易懂。这支持高效的手动测试执行。
Using unambiguous and precisely defined keywords allows the option to select whether the execution of a test case is done manually, or is done with automation. In the case of automation, it is expected that the keywords will be implemented as keyword-scripts. ---使用明确且精确定义的关键字，可以选择是手动执行测试用例，还是自动执行测试用例。在自动化的情况下，预期关键字将作为关键字脚本实现。
Testers working at the business level do not require technical understanding of the test automation framework to be able to create and edit test cases. ---在业务级别工作的测试人员不需要对测试自动化框架的技术理解，就可以创建和编辑测试用例。
Testers working at the technical level can implement or perform keyword-driven test cases, even if they have limited or no understanding of the business domain. ---在技术级别工作的测试人员可以实现或执行关键字驱动的测试用例，即使他们对业务领域的理解有限或没有。
Testers on a technical level can implement test cases using a language that is understandable to domain experts and that can be reviewed by them for business correctness.
